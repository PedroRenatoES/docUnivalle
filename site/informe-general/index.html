    1 Introducción

    Nuestro proyecto tiene como objetivo desarrollar una plataforma innovadora que facilite a los estudiantes la localización de objetos perdidos dentro de la universidad, promoviendo una comunidad más conectada y eficiente. Este sistema no solo busca simplificar la recuperación de pertenencias extraviadas, sino también fomentar un entorno de colaboración y responsabilidad entre los usuarios.

    2 Definición de Proyecto

    2.1 Descripción de la Problemática

    En el entorno universitario, la pérdida de objetos personales es un problema recurrente que afecta tanto a estudiantes como al personal académico. Desde mochilas, llaves, y dispositivos electrónicos, hasta documentos importantes, estas pérdidas generan no solo inconvenientes personales, sino también una pérdida significativa de tiempo y recursos en su recuperación. Actualmente, las herramientas o procesos disponibles para reportar y encontrar estos objetos son ineficientes o inexistentes, lo que limita la posibilidad de reubicar pertenencias extraviadas. Este escenario refleja la necesidad urgente de una solución tecnológica que centralice y facilite el proceso de localizar y devolver objetos perdidos.

    2.2 Objetivo General

    Desarrollar una plataforma digital que permita a los estudiantes y personal de la universidad reportar y buscar objetos perdidos de manera rápida, eficiente y accesible, fomentando la colaboración entre los usuarios y reduciendo significativamente el tiempo de recuperación de pertenencias extraviadas.

    2.3 Objetivos Específicos

    1. Diseñar e implementar un sistema intuitivo que permita a los usuarios registrar objetos perdidos o encontrados en tiempo real.
    2. Desarrollar una base de datos centralizada y segura para gestionar los reportes de manera eficiente y confiable.
    3. Implementar herramientas de búsqueda avanzadas que faciliten a los usuarios localizar objetos específicos según criterios como fecha, ubicación y tipo de objeto.
    4. Diseñar una interfaz amigable y accesible para promover una experiencia de usuario positiva y fomentar el uso continuo de la plataforma.
    5. Promover una cultura de responsabilidad y colaboración dentro de la comunidad universitaria mediante el uso de la plataforma.

    3 Especificación de Requerimientos

    3.1 Alcance

    El alcance de este proyecto está limitado al ámbito de la universidad, específicamente enfocado en los estudiantes, docentes, y personal administrativo de la misma. La plataforma se diseñará para que todos los miembros de la comunidad universitaria puedan reportar, buscar y recuperar objetos perdidos dentro del campus universitario.

    3.2. Requerimientos Específicos

    3.2.1. Requerimientos Funcionales

      1. Registro de objetos perdidos: Los usuarios deben poder ingresar información detallada sobre los objetos perdidos, incluyendo descripción, fecha, ubicación, y cualquier otra característica relevante.
      2. Registro de objetos encontrados: Los usuarios deben tener la opción de registrar objetos encontrados, especificando la misma información que en el caso de los objetos perdidos.
      3. Búsqueda de objetos: La plataforma debe permitir a los usuarios realizar búsquedas, para localizar pertenencias perdidas de manera eficiente.
      4. Interfaz de usuario: La plataforma debe contar con una interfaz de usuario amigable e intuitiva, tanto para el registro como para la búsqueda de objetos.
      5. Gestión de la base de datos: La plataforma debe contar con una base de datos centralizada que almacene todos los objetos perdidos y encontrados, accesible para los usuarios con los permisos adecuados.

    3.2.2. Requerimientos No Funcionales

    • Accesibilidad: La plataforma debe ser accesible desde dispositivos móviles y de web, con una interfaz adaptativa que garantice una experiencia de usuario adecuada en diferentes tipos de dispositivos.
    • Rendimiento: La aplicación debe cargar rápidamente, incluso con una gran cantidad de usuarios registrados y objetos reportados en la base de datos.
    • Escalabilidad: Aunque el alcance actual está limitado a la universidad, la plataforma va estar diseñada de manera que permita una futura expansión a otras instituciones si se decide hacerlo.
    • Fiabilidad: El sistema debe ser confiable, con alta disponibilidad y sin caídas frecuentes.
    • Usabilidad: La plataforma debe ser fácil de usar para los estudiantes, sin requerir conocimientos técnicos previos.
    • Compatibilidad: El sistema debe ser compatible con los navegadores más utilizados (Google Chrome, Mozilla Firefox, Microsoft Edge, Safari) y con el sistema operativo Android.

    3.3 Atributos de Producto

    1. Intuitividad: La interfaz es ser fácil de navegar, con un diseño claro y accesible para usuarios de diferentes edades y niveles tecnológicos.
    2. Seguridad: Los datos de los usuarios deben estar protegidos mediante medidas de seguridad como encriptación y autenticación de usuarios.
    3. Eficiencia: La plataforma debe ser rápida tanto en la búsqueda de objetos como en el registro y gestión de datos, para minimizar el tiempo de espera de los usuarios.
    4. Confianza: El sistema debe generar confianza en los usuarios mediante la transparencia en el proceso de registro y búsqueda de objetos, así como en la gestión de la comunicación entre usuarios.
    5. Adaptabilidad: La plataforma debe permitir futuras mejoras o cambios en la funcionalidad, como la inclusión de nuevas características o la expansión del sistema a otros ámbitos.
    6. Interactividad: La plataforma debe permitir la interacción entre los usuarios para facilitar la recuperación de objetos perdidos mediante mensajes y notificaciones en tiempo real.

      • Objetivo: Comprender a fondo a los usuarios y sus necesidades.

      • Realizar entrevistas y encuestas con estudiantes para conocer sus experiencias respecto a la pérdida de objetos.

      • Observar el entorno universitario, identificando lugares comunes donde se suelen perder objetos.
      • Crear perfiles de usuario basados en la información recopilada, destacando sus motivaciones y frustraciones.
      • Definición

    2. Objetivo: El objetivo de esta fase es sintetizar la información recopilada durante el proceso de investigación para identificar de manera precisa el problema central que se busca resolver, tomando en cuenta las necesidades y las dificultades de los usuarios finales.

    2.1. Clasificación de necesidades

    La clasificación de necesidades es un paso clave para garantizar que el desarrollo del proyecto esté alineado con las expectativas y necesidades de los usuarios. En este caso, las necesidades se agrupan en distintas categorías para poder abordarlas adecuadamente y establecer un enfoque claro en el diseño de la solución. A través de este proceso, se logrará entender mejor el problema y priorizar las funcionalidades que se van a desarrollar.

    • Objetivo: Agrupar y categorizar las diferentes necesidades identificadas a través de entrevistas y encuestas realizadas a los usuarios finales.
    • Actividades: Análisis de los datos de entrevistas y encuestas:

    Se revisarán y analizarán las respuestas de los usuarios para identificar patrones comunes en las necesidades, frustraciones y deseos. Esto incluirá la categorización de las respuestas en temas comunes (ej. accesibilidad, facilidad de uso, tiempos de respuesta, etc.).

    Entregables:

    1. Informe de clasificación de necesidades: Un documento que organiza las necesidades en categorías y proporciona una priorización según su importancia y urgencia.
    2. Declaración del problema: Un enunciado que resuma el problema identificado a partir de la investigación de usuarios.
    3. Ideación
      • Objetivo: Generar una amplia variedad de ideas y posibles soluciones al problema definido.
      • Actividades:
    4. Realizar sesiones de brainstorming donde todos los miembros del equipo puedan proponer ideas sin restricciones.
    5. Utilizar técnicas como “brainstorming” o “mapas mentales” para fomentar la creatividad.
    6. Seleccionar las ideas más prometedoras y relevantes, dándoles prioridad según su viabilidad y potencial impacto.
    7. Prototipado
      • Objetivo: Convertir las ideas seleccionadas en prototipos tangibles que puedan ser evaluados.
      • Actividades:
    8. Crear versiones simples de la plataforma utilizando herramientas de prototipado rápido (prototipos digitales).
    9. Desarrollar una interfaz básica en Dart que permita a los usuarios interactuar con la solución de manera preliminar.
    10. Preparar escenarios de uso para mostrar cómo funcionará la plataforma en la práctica.
    11. Pruebas
      • Objetivo: Validar las soluciones desarrolladas y recoger retroalimentación para futuras mejoras.
      • Actividades:
    12. Realizar pruebas de usabilidad con estudiantes, observando cómo interactúan con el prototipo.
    13. Recoger comentarios sobre la experiencia del usuario, identificando áreas de mejora.
    14. Iterar sobre el diseño basado en la retroalimentación, ajustando la plataforma para abordar las necesidades y expectativas de los usuarios.

    9 Metodología Programación Extrema (XP)

    9.1. Prácticas de XP

    Para asegurar un desarrollo ágil y de alta calidad, el equipo adoptó las siguientes prácticas de Extreme Programming (XP):

      1. Pair Programming: José y Renato trabajaron en pareja para mejorar la calidad del código, minimizando errores y asegurando un entendimiento común del proyecto.
    • Pruebas Manuales: Se implementaron pruebas manuales al final de cada sprint para asegurar que la plataforma cumpla con las expectativas del usuario.
    • Integración Continua: El equipo realizó integraciones diarias del código para evitar problemas de fusión y asegurar que la plataforma esté siempre actualizada.
    • Retroalimentación Rápida: Se llevaron a cabo reuniones diarias para ajustar el plan de desarrollo según el progreso y el feedback, permitiendo adaptaciones ágiles a las necesidades del proyecto.

    10 Módulo de Procesamiento de Datos – Flujo

    1. Ingreso al Sistema

      Este flujo comienza con la autenticación del usuario en la plataforma. Se requiere que los usuarios ingresen sus credenciales (correo electrónico y contraseña).

    2. Flujo principal:

        1. El usuario accede a la página de inicio de sesión.
      • Ingresa sus credenciales y se verifica su identidad.
      • Si las credenciales son correctas, el sistema redirige a la página principal.

    Manejo de errores: Si las credenciales son incorrectas, se muestra un mensaje de error con recomendaciones para restablecer la contraseña. 1. Interacción con la Sección de Analíticas

    La aplicación de objetos perdidos está diseñada para proporcionar análisis detallados a los administradores mediante una sección de Dashboard. Al acceder con una cuenta de usuario administrador, el sistema redirige automáticamente al panel de control, donde se visualizan diversos gráficos y estadísticas. En esta sección, se presentan los siguientes datos clave:

    • Usuarios por carrera: Un gráfico que muestra la distribución de los usuarios de la aplicación según su carrera en la universidad.
    • Objetos perdidos por mes: Un gráfico de barras que permite observar las pérdidas registradas en cada mes, ayudando a identificar tendencias estacionales.
    • Categoría más perdida: Un gráfico de sectores que presenta las categorías más comunes de objetos perdidos, facilitando el análisis de patrones en el tipo de objetos reportados.
    • Ubicaciones con más pérdidas: Un mapa interactivo o gráfico que destaca las ubicaciones dentro de la universidad donde se han registrado más objetos perdidos, lo cual permite a los administradores entender mejor los puntos críticos de la institución.


    10.3. Uso del Módulo de Procesamiento de Datos

    El módulo de procesamiento de datos en esta aplicación tiene como objetivo transformar y organizar la información que proviene de los usuarios y las interacciones con la plataforma para generar las analíticas presentadas en el Dashboard. Este módulo se encarga de filtrar, ordenar y agrupar los datos, como los objetos perdidos registrados, las estadísticas de usuarios, y las interacciones dentro de la aplicación. A través de algoritmos y procesos de backend, el sistema organiza la información de manera eficiente para poder presentarla en los gráficos y tablas del Dashboard, asegurando que los datos estén actualizados y sean relevantes para la toma de decisiones del administrador.


    10.4. Funciones Adicionales para Administradores

    En el Dashboard, los administradores tienen acceso a un conjunto de funciones adicionales a través de un navbar que les permite gestionar la plataforma. Estas funcionalidades incluyen:

    • Gestión de usuarios: Acceso a una pantalla donde el administrador puede ver la lista completa de usuarios registrados en la aplicación, con la opción de eliminarlos o editarlos si es necesario.
    • Gestión de publicaciones y comentarios: Una sección donde el administrador puede visualizar todas las publicaciones de objetos perdidos y los comentarios realizados por los usuarios, con la capacidad de eliminar o gestionar estos elementos.


    10.5. Resumen del Flujo de Usuario

    El flujo de usuario en la aplicación está diseñado para ser intuitivo y eficiente. Un usuario estándar inicia sesión en la aplicación y accede a la pantalla principal de home base, donde puede realizar publicaciones de objetos perdidos. Si el usuario es un administrador, la aplicación lo redirige directamente al Dashboard, donde tiene acceso a las herramientas de análisis de datos y gestión de usuarios y contenidos. El administrador puede navegar por los diferentes módulos del panel para gestionar el sistema de objetos perdidos, revisar estadísticas y realizar acciones de moderación sobre usuarios y publicaciones. En ambos casos, la interacción con la aplicación está diseñada para ser fluida y con un enfoque en la facilidad de uso.

    11 Tecnologías

    1. Flutter

    2. Justificación: Hemos elegido Flutter como framework de desarrollo porque permite construir aplicaciones multiplataforma (móvil y web) a partir de una única base de código. Esto optimiza el tiempo y los recursos necesarios para el desarrollo, ya que nos permitió lanzar tanto la versión móvil como la versión web de la plataforma sin duplicar esfuerzos. Flutter también ofrece una interfaz de usuario altamente personalizable y una amplia comunidad de soporte, lo que facilita la implementación de una UI atractiva y funcional en ambas plataformas.

    3. Dart

    4. Justificación: Dart es el lenguaje de programación utilizado por Flutter. Su rendimiento y compatibilidad directa con Flutter lo convierten en la elección ideal para este proyecto. Dart es eficiente en el manejo de estados y datos, lo cual es esencial para desarrollar una plataforma que requiere una navegación fluida y la gestión constante de información sobre objetos perdidos.

    5. MongoDB

    Justificación: MongoDB es nuestra elección para la base de datos debido a su flexibilidad y escalabilidad. Al ser una base de datos NoSQL, MongoDB permite un almacenamiento eficiente de datos en formato de documentos BSON (similar a JSON), lo que facilita la estructuración de datos complejos y variados sin la rigidez de un esquema fijo. Esto es especialmente útil para aplicaciones móviles donde los datos pueden cambiar o evolucionar rápidamente.

    1. Node.js y Express.js

    2. Justificación: En el desarrollo de nuestra aplicación, hemos adoptado Node.js como el entorno de ejecución principal debido a su rendimiento eficiente y su capacidad para manejar aplicaciones altamente escalables y en tiempo real. Complementando esto, utilizamos Express.js como framework para la creación de nuestros servicios backend, lo que nos permite estructurar de manera modular y clara las API necesarias para la interacción con la aplicación. Node.js nos proporciona un modelo basado en eventos y no bloqueante, ideal para manejar múltiples solicitudes simultáneamente, mientras que Express.js simplifica la gestión de rutas, middleware y controladores. Esta combinación tecnológica asegura la implementación de servicios web robustos, ágiles y preparados para futuras ampliaciones en la funcionalidad de la aplicación.
